COMO EXECUTAR
==========================================================================

Pré-requisitos: Python 3.10+, ~4GB RAM livre, DASHSCOPE_API_KEY ou Ollama.

1. Clonar repositório e instalar dependências
   git clone <repo>
   cd BluaDiagnostics
   pip install -r requirements.txt

2. Configurar variáveis de ambiente
   cp .env.example .env
   # Editar .env com sua DASHSCOPE_API_KEY

3. Popular ChromaDB com a knowledge base
   bash scripts/index_kb.sh

4. Iniciar a interface Dash (principal)
   python app/dash_app.py
   # Abre em http://localhost:8050

   OU interface Streamlit (fallback)
   streamlit run app/streamlit_app.py

5. Rodar evals Sprint 2
   python -m evals.run_evals_sprint2
   # Gera evals/sprint2_results.json + gráficos em docs/figures/

6. Rodar testes pytest
   pytest tests/ -v

SISTEMA — DESTAQUES
==========================================================================

ARQUITETURA:
- 10 nós LangGraph (vs 7 na Sprint 1)
- 5 agentes especializados (supervisor + checkup + triagem + suporte + prescricao)
- 2 nós de proteção (pre_safety + safety dupla camada)
- 1 nó determinístico de escalada (SAMU 192 / FAST AVC)
- Memory truncation summarize-and-replace
- Supervisor estatal (força triagem se RED_FLAG persistir)

RAG:
- ChromaDB persistente + embeddings multilingual-e5-large
- MMR para diversidade contextual
- Auto-RAG: reformulação clínica da query
- Cross-encoder reranker ATIVO no agente Triagem
- Filtro por categoria de metadado (red_flag, bula, protocolo, etc.)
- 11 documentos cardiovasculares na knowledge base

SAFETY:
- Pre-safety regex (<10ms) — bloqueia jailbreak/OOS óbvios
- Safety pós-resposta dupla camada (heurística + LLM auditor)
- Tag inviolável [RASCUNHO_AGUARDANDO_REVISAO_MEDICA] em 4 camadas:
  tool + prompt + safety check + critério de eval

EVALS:
- 32 casos cobrindo happy_path, red_flag (com apresentações atípicas),
  jailbreak, out_of_scope, prescricao
- LLM-as-judge automatizado
- Métricas: acurácia por categoria, taxa escalada red_flag,
  taxa tag rascunho prescricao, latência p50/p95, custo por agente
- 4 iterações de prompt documentadas (v1 → v4) com tabelas comparativas

INTERFACE:
- Dash com painel técnico em tempo real (RAG, trajetória, tools, safety)
- Confidence scoring numérico baseado em sinais reais
- HITL síncrono via LangGraph interrupt_before
- Indicador de backend (DashScope/Ollama) com troca via env var
- Alerta sonoro em red flag (alert.wav)
- Design system HUD clínico (CSS reutilizado do projeto cardiac_dashboard_dash)
- Fallback Streamlit para garantia

EXECUÇÃO TRI-MODAL:
- DashScope qwen-plus (cloud, default, ~$0.001/turno)
- Ollama qwen2.5:14b (on-prem, gratuito, narrativa LGPD)
- Híbrido (alternância via .env)

OBSERVABILIDADE (BÔNUS):

- Traces aninhados visíveis: supervisor → rag → agente → tools → safety

==========================================================================
ESCOPO E LIMITAÇÕES
==========================================================================

O BluaDiagnostics opera EXCLUSIVAMENTE em escopo cardiovascular. Esta é
uma decisão de design (segurança > generalidade) implementada em 4 camadas:
pre_safety, prompts dos agentes, lista branca da tool de prescrição, e
critérios de eval. Pedidos fora do escopo são polidamente recusados com
redirecionamento ao canal de clínica geral Care Plus.

O sistema NÃO substitui avaliação médica humana. Em emergências reais,
o usuário deve ligar para o SAMU (192). A tag inviolável de prescrição
garante que rascunhos nunca virem prescrição válida sem revisão médica.
